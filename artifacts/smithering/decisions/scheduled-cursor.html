E14 — Manual runs must not corrupt scheduled window continuity (eng doc §6.1, v3)

Context. Review (major finding 5): the design derives the next scheduled window from "prior run.window_end", and manual runs share the run table. A manual custom window could advance/overlap the scheduled lineage → gaps or overlaps against AC-2.1.

Decision

Scheduled windows derive from chat_config.sched_cursor, advanced only by the scheduled path (transactionally with the D2 claim). Manual runs never touch it. If a scheduled claim conflicts with an existing row for the identical window (e.g. a manual run already covered exactly it), the cursor still advances — window already handled, no duplicate, no gap. Manual windows overlapping (not matching) scheduled ones are allowed and badged in the UI; AC-2.1 governs the scheduled lineage.

Example (the bug this kills)

Hourly schedule, cursor at 14:00.
Operator manually summarizes [14:30, 14:45).
v2: next scheduled window computed from prior run.window_end = 14:45
    → scheduled summary covers [14:45,15:00) — messages 14:00–14:30 NEVER summarized. ✗
v3: sched_cursor still 14:00 → scheduled run covers [14:00,15:00) as required. ✓
    (14:30–14:45 appears in both; manual overlap badge shown.)

Alternatives considered

OptionVerdictWhy
Derive from last scheduled run row (filter by trigger)RejectedWorks until a failed/deleted scheduled row breaks the chain; a cursor column is O(1), race-free with the claim txn, and survives lineage queries.
Forbid manual windows overlapping the scheduleRejectedBreaks AC-7.2's "chosen window"; operator intent should win with disclosure, not be blocked.
Partial unique index per trigger (allow duplicate windows)RejectedWould permit duplicate summaries for one window, violating AC-5.3.
sched_cursor watermark + conflict-advances ruleChosenGapless/overlap-free scheduled lineage by construction; property-testable over random interleavings.
