E1 — Exactly-once summary delivery over a non-idempotent API

Context. AC-5.3 forbids duplicate summary posts; AC-9.4 forbids duplicated side effects on resume. Telegram sendMessage has no idempotency key (probe A1). The v1 design (record telegram_message_id after a successful send) fails on the crash cut after provider-ack, before ledger write, and on concurrent executors both seeing an unsent chunk.

Decision

Two mechanisms, layered:

  1. Single writer — a per-run lease with a fencing token (run.lease_owner, see E2); every run_chunk write is fenced, so a concurrent/stale executor's reserve write hits zero rows before it can call Telegram.
  2. Two-phase send per chunk: reserving (write-ahead intent) → send content-free placeholder "…" → record its message_id (reserved) → editMessageText to the real chunk text (edited, idempotent to repeat) → sent.

Crash-cut table (the core of the argument)

Crash between…On resumeWorst outcome
start → reserving writeredo from scratchnone
reserving → placeholder acksend new placeholdernone (nothing was sent)
placeholder ack → recordsend new placeholder, continueone orphan "…" message — content-free
record → editedit recorded idnone
edit ack → edited/sent writere-edit; "message is not modified" = successnone

Residual — bounded and escalated (v3 revision): a stray "…" placeholder in the rare ack/record cut. Duplicate summary content is impossible by construction. The orphan cannot be deleted (its id was never learned) — the irreducible floor given no idempotency key. Bound: one orphan per attempt crossing the cut, so ≤ max_attempts (5) per run — the invariant test asserts orphans ≤ attempt_count, not "at most one". Escalation: strictly read, REQ-5's "exactly once and intact" and the no-spam stance do not admit even a content-free stray, so this residual is a declared deviation gated on human acceptance (assumption A13 in eng doc §16). Proposed AC-5.3 amendment: "exactly-once applies to summary content; content-free placeholders from crash-during-send are permitted, bounded by the per-run attempt limit." §5 implementation does not start until accepted. The same cut also applies to the E8 disclosure greeting (at-least-once, once per crashed attempt), folded into the same acceptance ask.

Alternatives considered

AlternativeWhy rejected
Placeholder + edit (chosen)Only content-free op can double; edit is idempotent; fully testable via fault injection.
Record-after-send only (v1)Duplicates full chunk content on the ack/record cut and under executor races — the review's exact objection.
Mark run failed on ambiguous cut, require operator replayViolates AC-5.2 ("posts without manual intervention").
Reconcile by reading chat historyBot API cannot list/read a chat's messages; bots don't receive their own sends via webhook.
Embed idempotency marker in content, dedupe laterNo read path exists to ever observe the marker (same reason as above).

Example (3 chunks, crash after chunk 2's placeholder ack)

chat timeline:            ledger (run_chunk):
[…]→edited to chunk1      #0 sent  id=101
[…]  ← orphan (id lost)   #1 reserving  ← crash here
                          resume: new placeholder id=103, edit → chunk2
[…]→edited to chunk2      #1 sent  id=103
[…]→edited to chunk3      #2 sent  id=104
Result: chunks 1,2,3 delivered exactly once + one stray "…".

Tests: fault-injected kill at every cut × every chunk index; repeated ambiguous cut across N attempts of the same chunk (orphans ≤ N, content still exactly once); concurrent double-executor race; invariant = delivered content multiset ≡ chunk list, orphan placeholders ≤ attempt_count, run ends posted. Red-before-green against the v1 logic.
