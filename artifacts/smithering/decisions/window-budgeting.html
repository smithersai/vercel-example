E5 — Large-window input strategy & E7 — tick scale envelope

E5: what happens when a window won't fit one prompt

Policy tiers (constants in one config module, boundary-tested):

Window sizePathUser-visible effect
≤ 80k est. tokenssingle Summarizer callnormal summary
≤ 20 segments × 60kmap-reduce: per-segment structured summaries → one reduce; participants/links passed verbatim so grounding holdsnormal summary
beyond one reduce levelrecursive (hierarchical) reduce, no level cap — intermediates grouped and reduced until one call remains; paginated fetch (5k rows/page), no message capsummary opens "High-volume window (N messages): condensed digest." — a mode disclosure, not a loss

v3 revision (review finding 3): the v2 tier "keep newest up to cap + warning" dropped covered messages — a visible but still lossy mode contradicting AC-1.1/AC-2.1, which admit no lossy exception. Replaced with unbounded recursive reduce: every message is read by exactly one map call at any volume; volume now costs time/tokens, never coverage. The lease/heartbeat model (E2) plus the >20-minute Sandbox path (REQ-9) are exactly what make arbitrarily long summarize runs safe, so "unbounded latency" is no longer a valid rejection — that earlier rejection is superseded here.

Alternatives rejected: unbounded single prompt (fails at scale); newest-N truncation with warning (v2 — lossy, violates AC-1.1; would require a PRD amendment the product doesn't want); silent truncation (worse); PRD amendment to permit loss (rejected: recursive reduce achieves the requirement as written).

Tests: exact-budget and budget+1 boundaries; planted-topic coverage across 3-segment and forced 2-level fixtures (first and last segment topics must surface); pagination fuzz around page boundaries; 50k-message benchmark with asserted wall/token ceilings; AC-3.3 grounding eval over map-reduce outputs.

E7: bounding the 1-minute tick globally

Envelope: 500 enabled chats, 50 concurrent runs, tick < 10 s — enforced by a load test, degrading gracefully beyond (backlog drains, never drops — AC-2.5).

  • Global TICK_CLAIM_CAP=25 per tick (on top of D25's per-chat 20); unclaimed windows stay due.
  • Fair scan cursor chat_config.last_claim_scan_at — no chat starvation, partial ticks resume.
  • MAX_CONCURRENT_RUNS=50 gate: claims proceed, invocations defer; the watchdog also sweeps stranded pending claims so deferral is safe.
  • Pooled Neon connections; set-based tick queries (probe A5 validates the pool).

Alternatives rejected: uncapped fan-out (v1 — can blow Function runtime/connection limits in an outage backlog); a queue service for backpressure (violates single-Postgres-store REQ-8); collapsing backlog windows into one (violates AC-4.5 per-window independence, already rejected in D25).

Test: seed 200 chats × 10 missed windows; controlled-clock ticks must drain monotonically within caps, no duplicate claims, bounded staleness, tick wall-time asserted.
