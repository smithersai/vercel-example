E10 — How smithers actually runs on Vercel (eng doc §2, v3)

Context. Adversarial review (critical finding 1): the v2 eng doc built a bespoke Postgres state machine and never specified how smithers runs serverlessly — but REQ-8's whole point is proving smithers on Vercel.

Decision

  • E10.1 The summarize pipeline is a smithers workflow; the Sandbox executor runs the smithers runtime in-process, per run, then exits. No resident daemon.
  • E10.2 Smithers persists workflow/run/step state via a Postgres state driver into a smithers schema in the same Neon DB; app run.smithers_run_id joins the two. If the driver doesn't exist / isn't serverless-safe, it is built upstream in smithers first (blocking probe A12).
  • E10.3 The operator UI = custom smithers Gateway UI as Next.js serverless routes reading app + smithers schemas from Postgres. The local bun gateway.ts process is dev-only, never deployed.
  • E10.4 UI Replay / watchdog retry = re-invoke the Sandbox executor for the same run.id, which resumes the same smithers run from the first non-completed step. One resume path for AC-7.3 and AC-9.4.

Alternatives considered

OptionVerdictWhy
Bespoke state machine only, smithers just for dev workflows (v2 status quo)RejectedFails the PRD problem statement — the template would prove nothing about smithers on Vercel.
Long-lived smithers gateway on a VM/containerRejectedViolates REQ-8/AC-8.1 (no operator-owned long-lived process).
Smithers state in SQLite on Sandbox ephemeral diskRejectedViolates AC-8.2 (single Postgres store); state dies with the sandbox.
Per-run in-process smithers runtime + Postgres driver + serverless Gateway read-modelChosenOnly shape satisfying AC-8.1 + AC-8.2 + REQ-9 simultaneously; step durability subsumes D23's hand-rolled checks.

Example flow

tick → INSERT run (claim) → invoke Sandbox(run.id)
Sandbox: acquire lease → smithers.resume(run.smithers_run_id ?? create)
  step assign      ✓ (done, skipped on resume)
  step threshold   ✓
  step summarize   ← crash here → watchdog re-invokes → resumes AT summarize
  step deliver (E1 protocol inside)
  step finalize → status='posted'

Risk / probe

A12 (blocking): 3-step spike workflow, killed after step 2, must resume in a fresh process against Postgres executing only step 3. Failure ⇒ upstream smithers Postgres-driver milestone precedes app implementation.
