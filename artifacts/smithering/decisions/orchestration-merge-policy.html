O2 — Merge policy: serialized merge lane; optimistic merging with postsubmit eviction is NOT built

Context: 06-orchestration.md O2 · 2026-07-01 · vercel-example implementation run (23 tickets, dep graph in artifacts/smithering/tickets.json).

Decision

Work lands through a single serialized merge lane: Smithers MergeQueue used purely as a concurrency limiter with limit 1. Per ticket: rebase onto main → rerun the full Tier-1 gate set on the rebased tree → fast-forward/squash into main → release slot. main is never red by construction.

Alternatives considered

OptionWhat it needsVerdict
Serialized lane (chosen)MergeQueue(limit=1) — exists today. Rebase+retest is worker logic already required.Zero new infrastructure; correctness by construction.
Optimistic merge + postsubmit evictionCustom CheckSuite wiring + Saga compensation + jj revert eviction logic — none of this exists in Smithers; MergeQueue is only a concurrency limiter, it does not evict.Rejected: negative ROI (see math), adds a bad-eviction failure class to a repo whose product is correctness gates.
Queue limit 2 with post-land re-runSmall config change.Held as the recorded escalation if the lane measurably stalls; not the default.

Example throughput math (why optimistic loses here)

Tier-1 gate set (fixture LLM, fake Telegram, local PG): ~5–10 min per ticket.
Workers: 3 (O5). Worst case all 3 finish simultaneously:
  serialized lane latency = 2 × 10 min of queueing per day-ish event.
Optimistic eviction saves that ~20 min but costs:
  eviction saga build+test (days), and every eviction rewrites main history
  mid-run while other worktrees auto-rebase — including PAST FAILED rebases
  (known smithers worktree resume behavior), i.e. silent breakage risk.
Load-bearing fact: smithers worktree runs auto-rebase on resume even if the rebase failed. Optimistic merging multiplies rebase events; serialization minimizes them. The O1 resume protocol (worker must run jj st/conflict check as step zero) is the belt for the residual.

Consequences

  • Landing is O(1) tickets at a time; development stays parallel (3 worktrees).
  • A Tier-1 failure in the lane returns the ticket to its worker with the output — no revert ever needed on main.
  • Tier-2 (Preview e2e, post-land) failures halt dispatch and open a fix ticket; they do not trigger reverts either unless the human orders one.
