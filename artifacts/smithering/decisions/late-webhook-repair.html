E9 — Late Telegram webhook delivery must not create summary gaps (eng doc §7, v3)

Context. Review (critical finding 2): Telegram retries webhook delivery for up to ~24h. v2 selected run input by sent_at ∈ [window_start, window_end); a message ingested after its window posted would be stored but never summarized — a permanent gap violating AC-1.1/AC-2.1.

Decision

  1. Ingest grace: a boundary b is due only at b + INGEST_GRACE (120s). Absorbs jitter; boundaries unchanged.
  2. Assignment ledger: run input = non-bot messages with assigned_run_id IS NULL AND sent_at < window_end, assigned (fenced) as the first workflow step. Skipped runs assign too (preserves AC-4.5).
  3. Repair: a message ingested after its covering window finalized stays unassigned → next run includes it under an explicit “Earlier (delivered late)” line, counted in run.late_message_count.

Example

Window W = [14:00,15:00) posts at 15:02.
15:40 Telegram finally delivers msg M (sent_at 14:37).
M inserts with assigned_run_id NULL.
16:00 run: input includes M → summary ends:
  "Earlier (delivered late): Alice shared the deploy checklist (14:37)."

Alternatives considered

OptionVerdictWhy
Long close-delay (e.g. 30 min) onlyRejectedDelays every summary; still loses >delay-late deliveries (retries can span hours).
Reopen/edit the already-posted summaryRejectedRe-summarizing a posted window collides with AC-5.3 exactly-once; edit churn is spammy.
Telegram getUpdates cursor reconciliationRejectedgetUpdates and webhook are mutually exclusive modes; can't run both.
Grace + unassigned-message repair with late flagChosenNo gaps at any lateness, no reopening, honest attribution; one query change.

Recorded interpretation. AC-1.1's "next summary whose window covers it" is read, for delayed delivery only, as "next posted summary, flagged late" — the alternative violates AC-5.3. Red-before-green: the repair e2e provably fails against v2's window query.
