Decision: smithers workflow state in the app's Neon Postgres — probed first, fixed upstream if broken

2026-07-01 · docs/planning/03-eng.md §1.5, probe A4 · serves AC-8.2

What was decided

Smithers' own run/workflow state uses the same Neon Postgres database as the app tables (chat, message, run, run_chunk) — no SQLite file, no second store, in any preview/production environment. Because this is the one integration nobody has done before (the PRD says so explicitly), it gets a blocking probe (A4): a 3-step workflow in a Sandbox, killed mid-run, must resume from Postgres alone. Any gap found is fixed upstream in /Users/williamcory/smithers (we maintain it), never worked around locally.

Alternatives considered

OptionWhy rejected / accepted
Same Postgres, smithers tables alongside app tables (chosen)AC-8.2 requires it verbatim ("all persistent state … in the single Postgres database"); also makes "point a fresh deploy at the DB and everything resumes" literally true.
Separate Postgres database for smithers stateStill "Postgres-only" in spirit, but breaks the one-database restore guarantee and doubles the Neon-branch-per-preview plumbing; no benefit.
SQLite in the Sandbox (smithers default)Sandbox filesystem is ephemeral — state dies with the invocation, killing crash-resume (AC-8.1/9.4); explicitly banned by PRD non-goals.
Skip smithers for the bot's own runs; use it only for dev workflowThe pattern is the product — a template that doesn't run smithers serverlessly proves nothing to the adopter audience.

Example: what probe A4 must show

1. smithers up summarize.tsx  (Postgres backend, inside Sandbox)
2. step1 completes → row in smithers state tables
3. kill -9 the sandbox process after step1
4. re-invoke with same runId
   ✅ pass: step2/step3 run, step1 NOT re-executed, no *.db file exists
   ❌ fail: smithers bug → fix in /Users/williamcory/smithers + tests, re-probe
