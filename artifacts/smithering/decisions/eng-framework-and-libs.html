Decision: Next.js app, thin Telegram client, croner, Vitest+Playwright, structured LLM output

2026-07-01 · docs/planning/03-eng.md §2 · resolves the "open" items from eng-deps §7

What was decided (and the alternative each beat)

PickBeatWhy
Next.js on Vercel (routes + operator UI in one app)Separate SPA + bare FunctionsAC-7.6 wants UI in the same deployment; one framework = one coverage config, one deploy, template stays copyable.
Thin Telegram client (raw fetch behind the 3-method C5 interface)telegraf / full grammY frameworkD26/D27's retry+ledger idempotency must be ours; a framework's built-in retry/session machinery would fight the Postgres-driven model. Matches eng-deps §5 recommendation.
croner for D4's cron walkcron-parserNative TypeScript + built-in IANA tz handling; DST correctness gated by blocking probe A5, and it's swappable behind C6's one-function interface if the probe fails.
Vitest (v8 coverage) + PlaywrightJest / Cypress / node:testVitest branch coverage is the 100% gate mechanism (AC-10.1); Playwright runs one e2e suite against both localhost and Preview URLs (AC-10.2/10.3) — the "same suite, two targets" requirement is its core strength.
LLM returns structured JSON; MarkdownV2 rendered by our deterministic code (claude-sonnet-5)LLM emits final MarkdownV2 directlyEscaping/chunking become 100%-coverable pure functions (AC-3.4, D8) and groundedness eval (AC-3.3) checks structured fields, not prose. LLM-emitted markup would make the coverage gate depend on live LLM calls.
