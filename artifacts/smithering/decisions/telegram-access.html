E12 — Making sure the bot actually receives ordinary group messages (eng doc §13, v3)

Context. Review (major finding 8): Telegram bots default to privacy mode — they receive only commands/replies in groups. All fakes pass while a real deployment silently summarizes nothing. REQ-1 hinges on this and v2 never mentioned it.

Decision — three gates

  1. Setup gate (README checklist, amends D20): BotFather /setprivacy → Disable before adding the bot to chats (privacy changes apply on re-add); setWebhook with allowed_updates:["message","my_chat_member"] + secret_token.
  2. Automated check: pnpm check:telegram (post-deploy step + operator-UI red banner) fails unless getMe.can_read_all_group_messages === true, webhook URL matches the deployment, allowed_updates includes message, and getWebhookInfo.last_error_date isn't recent.
  3. Live smoke (extends D30): assert a plain human-sent group message lands in message within 60s. Bots cannot see other bots' messages, so this leg is prompted/human-in-the-loop — a platform constraint, recorded, not hidden; the getMe/getWebhookInfo legs run unattended nightly.

Failure mode this converts

Before: deploy ✓, e2e ✓ (fakes), bot in chat… every window skipped
        "below_threshold: 0 < 3" forever. Silent product failure.
After:  check:telegram fails loudly pre-traffic:
        "can_read_all_group_messages=false — run /setprivacy Disable and re-add the bot."
