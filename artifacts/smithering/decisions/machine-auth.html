E11 — Auth for Telegram/Cron callers; D18 scoped to the operator surface (eng doc §12, v3)

Context. Review (major finding 7): D18 says "all server routes reject requests without the UI token/cookie" — impossible for Telegram and Vercel Cron callers, and v2 specified no auth for them at all (anyone could POST fake messages or force ticks).

Decision

SurfaceCredentialMechanism
Operator UI + operator APIUI_TOKEND18 bearer→cookie (unchanged)
/api/telegram/webhookTELEGRAM_WEBHOOK_SECRETsecret_token set at setWebhook; Telegram echoes it as X-Telegram-Bot-Api-Secret-Token; constant-time compare, 403 otherwise
/api/cron/tickCRON_SECRETAuthorization: Bearer — Vercel injects it on cron invocations
/api/test/*E2E_TEST_ROUTES=1 AND UI_TOKENroute absent unless env flag set; never set in production

Three independent, rotatable, env-supplied secrets (AC-11.3). D18's wording gets a design-doc errata: it governs the operator surface only.

Alternatives considered

OptionVerdictWhy
Reuse UI_TOKEN for all three surfacesRejectedTelegram can't send it; a single secret leak would open every surface; rotation couples unrelated callers.
Webhook auth by Telegram source-IP allowlistRejectedIP ranges shift; unverifiable behind Vercel's proxy layers; secret_token is the platform-native mechanism.
No auth on tick ("it's idempotent anyway")RejectedUnauthenticated forced ticks enable DB-load abuse and watchdog manipulation.
Per-surface secrets as aboveChosenPlatform-native on both machine surfaces; cross-credential rejection is testable.

Tests. Each endpoint: 401/403 on missing credential, wrong credential, and the other surface's credential.
