E7 — Bounding the one-minute tick globally (eng doc §9)

Context. D21/D25 bound per-chat fan-out but not the global tick: 500 backlogged chats × 20 windows would try to claim 10,000 runs in one invocation.

Decision

  • Stated envelope: 500 enabled chats, 50 concurrent runs, tick < 10s — enforced by a load test, not aspirational.
  • TICK_CLAIM_CAP=25 global claims/tick; unclaimed windows stay due (correctness never depends on a tick finishing).
  • Fair scan cursor (last_claim_scan_at ASC) — no chat starvation; partial ticks resume.
  • Concurrency gate at MAX_CONCURRENT_RUNS=50; watchdog sweeps stranded pending rows.
  • Pooled connections; set-based queries only.

Alternatives considered

OptionVerdictWhy
Unbounded scan-and-claim (v1)RejectedTick wall-time and connection use unbounded under backlog; can exceed Function limits.
Per-chat cron entriesRejectedVercel cron entries are static config; AC-6.3 requires config changes without redeploy.
Global cap + fair cursor + concurrency gateChosenBounded tick regardless of backlog; drain guaranteed by AC-2.5 semantics ("due until claimed").
