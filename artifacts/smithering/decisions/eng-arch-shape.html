Decision: single 1-minute cron tick → DB claim → Sandbox executor

2026-07-01 · docs/planning/03-eng.md §1 · confirms and concretizes design decisions D2/D21–D23

What was decided

One Vercel Cron entry fires /api/cron/tick every minute. The tick only scans chat_config, claims due windows with INSERT INTO run … ON CONFLICT (chat_id, window_start, window_end) DO NOTHING, and triggers one Vercel Sandbox invocation per claimed run, passing only {runId}. The Sandbox re-reads all state from Postgres. The same tick doubles as watchdog for runs stuck in running.

Alternatives considered

OptionWhy rejected / accepted
Tick + claim + Sandbox fan-out (chosen)Only shape satisfying all three: no long-lived process (REQ-8), >20-min tasks (REQ-9), one Postgres store (AC-8.2). Matches Vercel's own documented cron pattern and pg-boss's singleton-key claim precedent (eng-oss §1.2, §2.1).
Do summarization inside the cron FunctionHits Function execution ceiling; violates AC-9.1's "no short-task shortcut" and can't reach 20 minutes.
Per-chat Vercel Cron entriesCron entries are deploy-time config — AC-6.3 requires schedule changes without redeploy; also per-plan cron-count limits.
Durable-execution vendor (Inngest/Trigger.dev/Temporal)Explicitly rejected by the PRD problem statement — state would live outside the user's Postgres; the whole point of the template is not needing one.
Rich invocation payload (window, config, messages) instead of just runIdA watchdog re-invocation has no access to the original payload; re-reading by runId makes retry = first-run, one code path (D22/D23).

Example: two concurrent ticks race on the same window

Tick A: INSERT run (chat 7, 14:00–15:00) → inserted 1 row → spawn Sandbox(run 42)
Tick B: INSERT run (chat 7, 14:00–15:00) → ON CONFLICT DO NOTHING → 0 rows → no spawn
Result: exactly one run, no app-level locking, verified by the AC-5.3 concurrency test.
